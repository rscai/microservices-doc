目标

本章中，我们将基于 Spring Boot 开发提供 RESTful API 的 Inventory 微服务。我们将学习：

  • 使用 Spring Web MVC 创建 RESTful Web 服务

  • 使用 Spring HATEOAS 创建遵循 HATEOAS 原则的 RESTful Web 服务

  • 使用 Spring Data/Spring Data JPA 访问关系式数据库（本例中使用 MySQL）

  • 使用 Eureka Discovery Client 注册服务

  • 使用 Spring Security 实现基于 URL 的访问控制

设计

用例

作为一个仓储运营，其可以：

  1. 新建仓储项，每个仓储项包含商品标识、单价、数量等属性；

  2. 入库，增加仓储项的数量；

  3. 出库，减少仓储项的数量。

Figure 1. 用例

业务模型

仓储业务主要涉及到两个业务模型：仓储项「InventoryItem」和仓储项数量变更「InventoryItemQuantityChange」。

InventoryItem

记录了仓储项的所有属性，包括商品标识、单价、数量等。

InventoryItemQuantityChange

仓储项的数量变更是比较频繁，多来源且要求可追遡可撤消。所以，将仓储项数量变更设计成独立的业务模型。

架构

整体採用网关模式。所有外部请求都被发送至网关，网关从发现仓储服务实例，并将请求转发给仓储服务实例。仓储服务实例在启动阶段将自身注册于 Eureka。

JPA

Java 持久化 API（JPA）是一个 Java 应用程式接口规范，描述了使用 Java 标准版平台（Java SE）和 Java 企业平台（Java EE）的应用中的关系数据的管理。

持久化，在这𥚃包括三个层面的意思：

  • API 本身，定义在 javax.persistence 包内

  • Java 持久化查询语言（JPQL）

  • 对象/关系元数据

在仓库模式「Repository Pattern」中，业务逻辑代码与仓库「Repository」介面交互存取业务实体。Repository 通过特定的 Data Mapper 实现业务实体与具体 Data Source 之间的映射。JPA就是一种针对关系式数据库 Data Source 的 Data Mapper。

关系数据库（英语：Relational database），是创建在关系模型基础上的数据库，借助于集合代数等数学概念和方法来处理数据库中的数据。现实世界中的各种实体以及实体之间的各种联系均用关系模型来表示。

— 关系数据库
https://zh.wikipedia.org/zh-cn/关系数据库

用于数据库管理的关系模型（英语：Relational model）是基于谓词逻辑和集合论的一种数据模型，广泛被使用于数据库之中。最旲于1970年由埃德加·科德提出。

关系模型的基本假定是所有数据都表示为数学上的关系，就是说 n 个集合的笛卡尔积的一个子集，有关这种数据的推理通过二值（就是说没有 NULL）的谓词逻辑来进行，这意味着对每个命题都有两种可能的赋值：要么是真要么是假。数据通过关系演算和关系代数的一种方式来操作。关系模型是採用二维表格结构表逹实体类型及实体间联系的数据模型。

关系模型允许设计者通过数据库规范化的提炼，去建立一个信息的一致性的模型。访问计划和其他实现与操作细节由 DBMS 引擎来处理，而不应该反映在逻辑模型中。这与 SQL DBMS 普遍的实践是对立的，在它们那𥚃性能调整经常需要改变逻辑模型。

基本的关系建造块是域或者叫数据类型。元组是属性的有序多重集（multiset），属性是域和值的有序对。关系变量（relvar）是域和名字的有序对（序偶）的集合，它充当关系的表头（header）。关系是元组的集合。尽管这些关系概念是数学上的定义，但它们可以宽松地映射到传统数据库概念上。表是关系公认的可视表示；元组类似于行的概念。

关系模型的基本原理是信息原理：所有信息都表示为关系中的资料值。所以，关系变量在设计时刻是相互无关系的；反而，设计者在多个关系变量中使用相同的域，如果一个属性依赖另一个属性，则通过参照完整性来强制这种依赖性。

数据库事务

数据库事务（简穪：事务）是数据库管理系统执行过程中的一个逻辑单位，由一个有限的数据库操作序列构成。

概要

数据库事务通常包含了一个序列的对数据库的读/写操作。包含有以下两个目的：

  1. 为数据库操作序列提供了一个从失败中恢复到正常状态的方法，同时提供了数据库即使在异常状态下仍能保持一致性的方法。

  2. 当多个应用程序在并发访问数据库时，可以在这些应用程序之间提供一个隔离方法，以防止彼此的操作互相干扰。

当事务被提供给了数据库管理系统（DBMS），则 DBMS 需要确保该事务中的所有操作都成功完成且其结果被永久保存在数据库中，如果事务中有的操作没有成功完成，则事务中的所有操作都需要回滚，回到事务执行前的状态；同时，该事务对数据库或者其他事务的执行无影响，所有的事务都好像在独立的运行。

ACID 性质

并非任意的对数据库的操作序列都是数据库事务。数据库事务拥有以下四个特性，习惯上被穪之为 ACID 特性。

  • 原子性（Atomicity）：事务作为一个整体被执行，包含在其中的对数据库的操作要么全部被执行，要么都不执行。

  • 一致性（Consistency）：事务应确保数据库的状态从一个一致状态转变为另一个一致状态。一致状态的含义是数据库中的数据应满足完整性约束。

  • 隔离性（Isolation）：多个事务并发执行时，一个事务的执行不应影响其他事务的执行。

  • 持久性（Durability）：已被提交的事务对数据库的修改应该永久保存在数据库中。

字段、访问器

JPA 本身仅是一个 API 规范，Spring Data JPA 採用 Hibernate 作为其 JPA 实现。

InventoryItem

InventoryItem.java
@Data
@Entity
@Relation(collectionRelation = "inventoryItems")
public class InventoryItem {

  @Id
  @GeneratedValue(generator = "system-uuid")
  @GenericGenerator(name = "system-uuid", strategy = "uuid")
  private String id;
  @Column(nullable = false, unique = true)
  private String productId;
  @Column(nullable = false)
  private int quantity;
  @Column(nullable = false, scale = 2)
  private BigDecimal unitPrice;
  @Column(nullable = false, updatable = false)
  @Temporal(TemporalType.TIMESTAMP)
  @CreationTimestamp
  private Date createdAt;
  @Column(nullable = false)
  @Temporal(TemporalType.TIMESTAMP)
  @UpdateTimestamp
  private Date updatedAt;
}

其使用 JPA 注解将实体 InventoryItem 映射至表 inventory_item，表名可以通过注解 @Table 显示指定，默认是将驼峰形式转换成以下划线连接单词的形式。字段也被一一映射至列。

id

String 类型的字段被映射为 varchar 类型的列，长度默认为255。列名及长度可通过注解 @Column 显示指定；使用注解 @Id 将该字段声明为主键；组合使用注解 GeneratedValue 和 GenericGenerator 声明以 UUID 策略生成该字段的值。在数据库实践中，主键值一般都是自动生成的。

product_id

String 类型的字段被映射为 varchar 类型的列，列名默认是把字段名转换成以下划线连接单词的形式。这𥚃显示用注解 @Column 声明该字段是不可为空且不可重复。

quantity

int 类型的字段被映射为 int 类型的列，长度默认为11。列名及长度都可通过注解 @Column 显示声明。

unitPrice

BigDecimal 类型的字段被映射为 decimal 的列，通过注解 @Column 声明小数点后的精度为2。

createdAt

Date 类型的字段可被映射为多种类型的列，比如 datetime、date 和 time。这𥚃使用注解 @Temporal 显示声明其映射为 datetime 类型的列。同时使用注解 @Column 声明其不可为空且不可更新。因为罗辑上讲，实体的创建时间戳在创建时就确定不变的。

updatedAt

使用注解 @Temporal 显示声明其映射为 datatime 类型的列。同时使用注解 @Column 声明其不可为空。

HAL

org.springframework.boot:spring-boot-starter-hateoas 提供了丰富的工具帮助应用开发者将 Java 对象编码为 HAL 文本及将 HAL 文本解码为 Java 对象。

相较与普通的 Java 对象， HAL 其实祗是附加了相关的链接「Link」。以 InventoryItem 为例，一个普通的对象编码为 JSON 仅包含其所有属性，例如：

{
  "id" : "ff8080817203fa06017203fa7df40011",
  "productId" : "P123456",
  "quantity" : 12,
  "unitPrice" : 12.12,
  "createdAt" : "2020-05-11T13:44:07.414+0000",
  "updatedAt" : "2020-05-11T13:44:07.414+0000"
}

但编码为 HAL+JSON 格式时则需附加一些链接，例如：

{
  "id" : "ff8080817203fa06017203fa7df40011",
  "productId" : "P123456",
  "quantity" : 12,
  "unitPrice" : 12.12,
  "createdAt" : "2020-05-11T13:44:07.414+0000",
  "updatedAt" : "2020-05-11T13:44:07.414+0000",
  "_links" : {
    "self" : {
      "href" : "http://localhost:8080/inventoryItems/ff8080817203fa06017203fa7df40011"
    }
  }
}

Spring Hateoas 提供了 EntityModel 和 PagedModel，分别用于将单个实体和分页查询结果「Page」包装成 HAL 模型。

1 Spring Data 提供了一个工具类 PagedResourcesAssembler 帮助应用开发者将 Page 转换为 PagedModel。
2 以构造器注入方式注入 EntityLinks。EntityLinks`可以帮助应用开发者构造单个实体的链接。因为所有链接都是绝对地址，其中包含了服务的域名端口等，构造过程比较繁琐。有了 `EntityLinks 的帮助可以大大减少应用开发者的工作量。
3 HAL 模型其实就是普通模型加链接。将单个实体的链接构造集中到一个补助方法中。
4 PagedResourcesAssembler` 可以直接从 Page 对象构造 PagedModel。
5 使用 EntityLinks 构造一个至实体自身的链接，该链接包含了服务域名和端口号。
