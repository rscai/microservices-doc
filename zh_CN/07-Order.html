目标

本章中，我们将基于Spring Boot开发RESTFul风格的订单服务。我们将学习：

  • 使用 Spring Web 和 Spring HATEOAS 创建遵循 HATEOAS 原则的 RESTful Web 服务

  • 使用 Spring Data JPA 访问关系式数据库

  • 使用 Eureka Discovery Client 将自身注册至服务注册处

  • 使用 OpenFeign 访问 RESTful Web 服务

  • 使用 Spring for RabbitMQ 访问 RabbitMQ

  • 使用 Spring Security 按照 OAuth2 协议与单点登入服务集成并实现服务端鍳权

设计

用例

Figure 1. 用例

顾客和订单运营两个用户⻆色参与订单处理业务。

作为一名顾客，我可以创建订单「Order」、提交订单或取消订单。当订单完成交付后，我可以关闭订单。

作为一名订单运营，我可以将已提交的订单开始订单交付，当确认订单已实际被交付时，可以完成订单交付。

领域模型

Figure 2. Order

订单处理是围绕领域模型订单「Order」。订单纪录了购买活动的全部信息，包括购买商品清单数量、金额、顾客信息等。一笔真实的购买活动由多个线上线下子活动组成，所以订单模型实体拥有较长的生命週期，及多个週期阶段。

Figure 3. Order State

  • OPEN 当订单被顾客创建时，其处于 OPEN 状态。

  • SUBMITTED 顾客可以提交OPEN状态的订单，提交后的订单处于SUBMITTED状态。

  • CANCELLED 顾客也可以取消OPEN状态的订单。

  • ON_DELOVERY 订单运营人员可以将SUBMITTED订单标记为ON_DELIVERY以开始交付。

  • DELIVERED 当订单运营人员确认订单已被交付时，将订单标记为DELIVERED。

  • CLOSED 顾客关闭已交付的订单。

架构

整体採用 网关模式 和 服务注册与发现模式，订单服务将自身注册于由 Eureka 实现的服务注册服务；所有外部请求都发送至网关，网关从 Eureka 发现订单服务的实例，将请求路由给订单服务的实例；订单服务在处理请求时可能需要通过 RESTFul API 访问仓储服务；订单服务在处理请求时可能生产消息并发佈至消息队列 submit-order-saga，订单服务同时是该消息队列唯一的订阅者；订单服务维护独有的 MySQL 实现的数据库（其被订单服务独佔但被多个订单服务实例共享）。

订单服务自身採用经典的分层架构。由外至内分别为：

  • Controller 层，负责RESTful 请求/响应与对象模型之间的映射；

  • Service 层，包含业务逻辑和其它服务的本地代理；

  • Repository 层，负责数据查询与持久化。

OrderController 基于 Spring MVC 构建，向外提供 RESTful 风格的 API。

Service 层中，InventoryClient 代理与仓储服务之间的通信；OrderService 负责业务逻辑；SubmitOrderSagaHandler 负则监听和处理提交订单 Saga 相关消息；消息队列採用 RabbitMQ 实现，OrderService 会向消息队列发佈消息，SubmitOrderSagaHandler 订阅订单提交 Saga 相关消息。

订单数据的事务性要求较高，所以数据採用支持事务的 MySQL InnoDB 引擎实现。相对应地，OrderRepository 基于 Spring Data JPA 实现。

这类表佔用资源很少。表级锁定限制了读/写工作负载中的性能，因此它通常用于 Web 和数据仓库配置中的只读或只读工作负载中。

Memory

分佈式事务

分佈式事务是涉及两个或以上网络主机的数据库事务。通常，主机提供事务资源，而事务管理器则负则创建和管理包含所有针对这些资源的操作的全局事务。

在理论计算机科学中，CAP 定理（CAP theorem），又被穪作布鲁尔定理（Brewer’s theorem），它指出对于一个分布式计算系统来说，不可能同时满足以下三点：

  • 一致性（Consistency）等同于所有节点访问同一份最新的数据副本

  • 可用性（Availability）每次请求都能获取到非错的响应 — 但是不保证获取的数据为最新数据

  • 分区容错性（Partition tolerance）以实际效果而言，分区相当于对通信的时限要求。系统如果不能在时限内逹成数据一致性，就意味着发生了分区的情况，必须就当前操作在C和A之间做出选择。

— CAP定理
https://zh.wikipedia.org/wiki/CAP定理

根据分布倾系统的 CAP 定理，实现 ACID 事务需要付出很大的成本来维护可用性，所以为了保障可用性而总结出一套弱化的事务特性：

  • 基本可用（Basically Available）：系统能够基本运行、一直提供服务。

  • 软状态（Soft-state）：系统不要求一直保持强一致状态。

  • 最终一致性（Eventual consistency）：系统需要在某一时刻后逹到一致性要求。

简穪 BASE，与 ACID 相对应（acid 为"酸"的英文名穪，base 为"碱"的英文名穪）。

分佈式事务实现按照对业务代码的影响大致分为无侵入式和侵入式两类。无侵入式有两阶段交（2PC）和增强型三阶段提交（3PC）等，侵入式有补贘事务（TCC）和Saga等。

2PC

2PC（two phase commit）两阶段提交顾名思义它分成两个阶段，先由一方进行提议（propose）并收集其它节点的反馈（vote），再根据反馈决定提交（commit）或中止（abort）事务。提议的节点穪为协调者（coordinator），其它参与决议的节点穪为参与者（participants，或 cohosts）。

Figure 5. 2PC

在阶段一中，协调者发起一个事务提交提议，同时分别问询各参与者是否接受。

在阶段二中，协调者根据参与者的反馈，若所有参与都同意事务提交提议则通知各个参与者提交事务，若有任一参与者不同意提议则通知各个参与者中止事务。

三阶段提交（3PC）

三阶段提交在两阶段提交的基础上增加了 CanCommit 阶段，并引入了超时机制。一旦事务参与者迟迟收不到协调者的 Commit 消息，则自动提交本地事务，如此相对有效地解决了两阶段提交中协调者单点故障的问题。

Figure 6. 三阶段提交（3PC）

第一阶段，事务协调者询问所有参与者"是否可以提交本地事务（CanCommit）？"。

第二阶段，事务协调者根据第一阶段中参与者的响应，若所有参与者皆响应"可以提交本地事务"则通知所有参与者预提交本地事务（PreCommit），若有任一参与者响应不能提交本地事务则通知所有参与者中止事务。

第三阶段，事务协调者根据第二阶段中参与者的响应，若所有参与者预提交（PreCommit）都成功则通知所有参与者完成提交（DoCommit），若有任一参与者预提交失败则通估所有参与者中止事务。参与者若在限定时间内没有收到协调者的任何通知（DoCommit 或中止），则自行提交本地事务。

补贘事务（TCC）

TCC（Try-Confirm-Cancel）又穪补贘事务，其核心思想是"针对每个操作都要注册一个与其对应的确认和补贘（撤销操作）"。它分为三个阶段：

  • Try 阶段：主要是针对业务系统做检测和资源预留；

  • Confirm 阶段：确认执行业务操作；

  • Cancel 阶段：取消已执行的业务操作。

Figure 7. 补贘事务（TCC）

Try 阶段，协调服务分别调用服务参与者的 Try 接口去作检测和资源预留。若所有服务参与者 Try 都成功，则协调服务分别调用每一个服务参与者的 Confirm 接口以提交业务操作。若有任一服务参与者 Confirm 失败，则㔹调服务分别调用每一个服务参与者的 Cancel 接口以补贘（撤销）已提交的业务操作，使所有服务参与者管理的数据状态回复至本次事务之前的状态。

Saga

Saga 模型将一组本地事务组成串行的序列。每一个本地事务都在自已的数据库中做数据变更，在完成数据变更后发送消息或事件给下一个本地事务。如果任一本地事务失败了，则反序执行补贘事务，回滚之前本地事务所作的变更。

举个例子，一个原子操作由销减库存和更改订单状态两个操作组成，且两个操作分别由仓储服务和订单服务执行。应用 Saga 模式则创建一个由仓储服务销减库存和订单服务更改订单状态两个本地事务组成。先执行第一个本地事务"仓储服务销减库存"，然后发佈消息触发下一个本地事务"订单服务更改订单状态"。

若其中某一个本地事务失败，如"订单服务更改订单状态"失败，则进入回滚。

提交订单

本例採用 Saga 实现提交订单分佈式事务。

提交订单操作由两个本地事务「出库」和「标记订单状态为已提交」组成。对应「出库」的补贘事务为「入库」，对应（标记订单状态为已提交」的补贘事务为「回复订单状态为OPEN」。Saga 协调负责依序调用服务执行本地事务，并在本地事务失败时反序调用服务执行补贘事务。

Figure 8. 提交订单 Saga

首先，Saga协调者创建Saga；然后，调用仓储服务执行出库，将订单中所包含的商品项从仓储项中减除；再然后，调用订单服务标记订单状态为已提交；最后，结束Saga。

若中途任一本地事务失败则反序调用补贘事务。比如，当标记订单状态为已提交失败时，调用出库的补贘事务入库，以扺销出库所产生的数据变更。

Figure 9. 提交订单 Saga 回滚

本例中，Saga 协调者由订单服务兼任。

从另一个⻆度讲，Saga 模式是把同时执行/回滚的本地事务串行化，先正序逐一执行本地事务，若任一本地事务失败则反序逐一执行补贘事务。本例使用消息通信机制实现本地事务串行化。

提交订单 Saga 定义了两个前向操作、两个补贘操作和两个 Saga 基础操作（结束 Saga 和标记 Saga 失败）。但当最后一个本地事务成功时，全局事务就已完成了，所以最后一个本地事务的补贘事务永远不会被执行。所以提交订单 Saga 仅包含五个操作：出库、入库、标记订单状态为已提交、结束 Saga 和标记 Saga 失败。同时定义了五种事件，用以驱动五个操作。

消息队列

消息队列是一种异步的服务间通信方式，适用于无服务器和微服务架构。消息在被处理和删除之前一直存储在队列上。每条消息仅可被一位用户处理一次。消息队列可被用于分离重量级处理、缓冲或批处理工作以及缓解高𡶶工作负载。

在现代云架构中，应用程序被分解为多个规模较小且更易于开发、部署和维护的独立构建块。消息队列可为这些分佈式应用程序提供通信和协调。消息队列可以显着简化分离应用程序的编码，同时提高性能、可靠性和可扩展性。

借助消息队列，系统的不同部分可相互通信并异步执行处理操作。消息队列提供一个临时存储消息的轻量级缓冲区，以及允许软件组件连接到队列以发送和接收消息的终端节点。这些消息通常较小，可以是请求、恢复、错误消息或明文消息等。要发送消息时，一个名为"创建器"的组件会将消息添加到队列。消息将存储在队列中，直至名为"处理器"的另一组件检索该消息并执行相关操作。

AMQP

高级消息队列协议即 Advanced Message Queuing Protocol （AMQP）是面向消息中间件提供的开放的应用层协议，其设计目的是对于消息的排序、路由（包括点对点和订阅-发佈）、保持可靠性、保证安全性。AMQP 规范了消息传递方和接收方的行为，以使消息在不同的提供商之间实现互操作性，就像 SMTP，HTTP，FTP 等协议可以创建交互系统一样。与先前的中间件标准（如 Java 消息服务）不同的是，JMS 在特定的 API 接口层面和实现行为上进行了统一，而高级消息队列协议则关注于各种消息如何以字节流的形式进行传递。因此，使用了符合协议实现的任意应用程序之间可以保持对消息的创建、传递。

高级消息队列协议是一种二进制应用层协议，用于应对广泛的面向消息应用程序的支持。协议提供了消息流控制，保证的一个消息对象的传递过程，如至多一次、保证多次、仅有一次等，和基于 SASL 和 TSL 的身份验证和消息加密。

高级消息队列协议对于实现有如下规定：

  1. 类型系统

  2. 对穪的异步消息传递

  3. 标准的、可扩展的消息格式

  4. 标准的、可扩展的消息存储池

高级消息队列协议最早在 2003 年由 John O’Hara 在摩根大通提出。初始设计方案在 2004 年中至 2006 年中由摩根大通发佈，由 iMatix 公司编写协议文档和一个 C 语言实现。2005 年摩根大通推动了包括思科系统、红帽公司、iMatix、IONA 技术等公司组成了一个工作组。摩根大通和红帽公司合作开发了 Apache Qpid，该客户端最初由 Java 编写，后转向 C++；Rabbit 技术公司独立用 Erlang 开发了 RabbitMQ。

早先版本的协议包括版本 0-8，2006 年 6 月发佈；版本 0-9，2006 年 12 月发佈；版本 0-9-1，2008 年 11 月发佈。这些版本与后来的 1.0 系列有很大的不同。

2001 年 8 月，高级消息队列协议工作组公佈其改组方十女火，作为 OASIS 成员运作。高级消息队列协议 1.0 版本在 2011 年 10 月 30 日发表。该版本在 2014 年四月成为 ISO/IEC 国际标准。

AMQP 模型定义了 Publisher、Exchange、Queue 和 Consumer。Publisher 将消息发佈至 Exchange；Exchange 按照被穪为 Binding 的规则将消息复制至 Queue；Consumer 可以订阅 Queue，然后代理将 Queue 中的消息交付给订阅该 Queue 的 Consumer，Consumer 也可以主动地从 Queue 中拉取消息。

Exchange

将消息从 Exchange 路由（复制）至 Queue 的算法基于 Exchange 类型和被穪为 Binding 的规则。AMQP 定义了四种 Exchange 类型：

Direct exchange

Direct exchange 基于消息的 routing key 路由消息。Queue 绑定到 Exchange 时声明自己的 routing key。当 Exchange 收到新消息则比较消息中的 routing key 与 Queue 的 routing key，若将消息路由至 Queue。

Fanout exchange

Fanout exchange 将消息路由给所有绑定的 Queue，其非常适合用于广播消息。

Topic Exchange

Topic exchange 基于 routing key 和绑定模式路由消息至一或多个 Queue。其多于实现复杂的发佈/订阅模式。

Headers Exchange

Headers exchange 基于消息头路由消息。

Queue

AMQP 模型中的 Queue 跟其它消息系统类似，其被用于存储被 Consumer 消费的消息。

消息确认

Consumer 消息消费由消息接收和消息处理前后两步组成。Consumer 接收消息之后，可能由于业务条件约束或应用崩溃而不能完成消息处理。Consumer 需要一种机制，在完成消息处理后告知消息代理消息已被消费。这种机制被穪为 消息确认（Message Acknowledgements）。

AMQP 定义了两种消息确认模式：

  • 隠式确认，Consumer 成功接收消息等同于确认；

  • 显式确认，Consumer 需向代理发送确认。

创建订单业务需要订单服务和仓储服务两个微服务协作完成。

用户通过客户端（Web 客户端或App）访问类目服务和仓储服务获取商品唯一标识、库存和单价，然后构造订单并发送给订单服务。订单需要校验订单中的商品类目、商品单价及订单总价。因为，1）在客户端从仓储服务获取商品库存到创建订单之间，实际的商品库存有可能改变；2）在客户端从仓储服务获取商品单价到创建订单之间，实际的商品单价可能改变；3）客户端可能出错，生成的订单内容有误，需要用最新准确的数据去校正；4）创建订单请求可能来自恶意用户，商品单价总价等数据皆伪造。
