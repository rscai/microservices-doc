目標

本章中，我們將基於Spring Boot開發RESTFul風格的訂單服務。我們將學習：

  • 使用 Spring Web 和 Spring HATEOAS 創建遵循 HATEOAS 原則的 RESTful Web 服務

  • 使用 Spring Data JPA 訪問關係式數據庫

  • 使用 Eureka Discovery Client 將自身注冊至服務注冊處

  • 使用 OpenFeign 訪問 RESTful Web 服務

  • 使用 Spring for RabbitMQ 訪問 RabbitMQ

  • 使用 Spring Security 按照 OAuth2 協議與單點登入服務集成並實現服務端鍳權

設計

用例

Figure 1. 用例

顧客和訂單運營兩個用戶⻆色參與訂單處理業務。

作為一名顧客，我可以創建訂單「Order」、提交訂單或取消訂單。當訂單完成交付後，我可以關閉訂單。

作為一名訂單運營，我可以將已提交的訂單開始訂單交付，當確認訂單已實際被交付時，可以完成訂單交付。

領域模型

Figure 2. Order

訂單處理是圍繞領域模型訂單「Order」。訂單紀錄了購買活動的全部信息，包括購買商品清單數量、金額、顧客信息等。一筆真實的購買活動由多個線上線下子活動組成，所以訂單模型實體拥有較長的生命週期，及多個週期階段。

Figure 3. Order State

  • OPEN 當訂單被顧客創建時，其處於 OPEN 狀態。

  • SUBMITTED 顧客可以提交OPEN狀態的訂單，提交後的訂單處於SUBMITTED狀態。

  • CANCELLED 顧客也可以取消OPEN狀態的訂單。

  • ON_DELOVERY 訂單運營人員可以將SUBMITTED訂單標記為ON_DELIVERY以開始交付。

  • DELIVERED 當訂單運營人員確認訂單已被交付時，將訂單標記為DELIVERED。

  • CLOSED 顧客關閉已交付的訂單。

架構

整體採用 網關模式 和 服務注冊與發現模式，訂單服務將自身注冊於由 Eureka 實現的服務注冊服務；所有外部請求都發送至網關，網關從 Eureka 發現訂單服務的實例，將請求路由給訂單服務的實例；訂單服務在處理請求時可能需要通過 RESTFul API 訪問倉儲服務；訂單服務在處理請求時可能生產消息並發佈至消息隊列 submit-order-saga，訂單服務同時是該消息隊列唯一的訂閱者；訂單服務維護獨有的 MySQL 實現的數據庫（其被訂單服務獨佔但被多個訂單服務實例共享）。

訂單服務自身採用經典的分層架構。由外至內分別為：

  • Controller 層，負責RESTful 請求/響應與對象模型之間的映射；

  • Service 層，包含業務邏輯和其它服務的本地代理；

  • Repository 層，負責數據查詢與持久化。

OrderController 基於 Spring MVC 構建，嚮外提供 RESTful 風格的 API。

Service 層中，InventoryClient 代理與倉儲服務之間的通信；OrderService 負責業務邏輯；SubmitOrderSagaHandler 負則監聽和處理提交訂單 Saga 相關消息；消息隊列採用 RabbitMQ 實現，OrderService 會嚮消息隊列發佈消息，SubmitOrderSagaHandler 訂閱訂單提交 Saga 相關消息。

訂單數據的事務性要求較高，所以數據採用支持事務的 MySQL InnoDB 引擎實現。相對應地，OrderRepository 基於 Spring Data JPA 實現。

MySQL

MySQL（官方发音为/maɪ ˌɛskjuːˈɛl/“My S-Q-L”，但也经常被错误读作/maɪ ˈsiːkwəl/“My Sequel”）原本是一个开放源码的关系数据库管理系统，原开发者为瑞典的MySQL AB公司，该公司于2008年被昇阳微系统（Sun Microsystems）收购。2009年，甲骨文公司（Oracle）收购昇阳微系统公司，MySQL成为Oracle旗下产品。

MySQL在过去由于性能高、成本低、可靠性好，已经成为最流行的开源数据库，因此被广泛地应用在Internet上的中小型网站中。随着MySQL的不断成熟，它也逐渐用于更多大规模网站和应用，比如维基百科、Google和Facebook等网站。非常流行的开源软件组合LAMP中的“M”指的就是MySQL。

但被甲骨文公司收购后，Oracle大幅调涨MySQL商业版的售价，且甲骨文公司不再支持另一个自由软件项目OpenSolaris的发展，因此导致自由软件社区们对于Oracle是否还会持续支持MySQL社区版（MySQL之中唯一的免费版本）有所隐忧，MySQL的创始人麦克尔·维德纽斯以MySQL为基础，成立分支计划MariaDB。而原先一些使用MySQL的开源软件逐渐转向MariaDB或其它的数据库。例如维基百科已于2013年正式宣布将从MySQL迁移到MariaDB数据库。

MySQL 的一大特性就是可插拔的存儲引擎架構。截止版本 8.0，MySQL 支持 10 種存儲引擎：

InnoDB

MySQL 8.0 的默認存儲引擎。InnoDB 是 MySQL 的事務安全（符合 ACID）存儲引擎，具有提交，回滾和崩潰焂復功能來保護用戶數據。InnoDB 行級鎖定（無需升級為更粗粒度的鎖定）和 Oracle 風格的一致非鎖定讀取可提高多用戶並發性和性能。InnoDB 將用戶數據存儲在聚集索引中，以減少基於主鍵的常見查詢的I/O。為了維護數據完整性，InnoDB 還支持 FOREIGN KEY 引用完整性約束。

MyISAM

这類表佔用資源很少。表級鎖定限制了讀/寫工作負載中的性能，因此它通常用於 Web 和數據倉庫配置中的只讀或只讀工作負載中。

Memory

將所有數據存儲在 RAM 中，以便在需要快速查找非關鍵數據的環境中進行快速訪問。

CSV

它的表實際上是帶有逗號分隔值的文本文件。CSV 表充許您以 CSV 格式導入或轉儲數據，以便與讀取和寫入相同格式的腳本和應用程序交換數據。由於 CSV 表未建立索引，因此通常在正常操作期間將數據保留在 InnoDB 表中，而僅在導入或導出階段使用 CSV 表。

Archive

這些緊湊的，未編制索引的表旨在用於存儲和檢索大量很少參考的歷史、存儲或安全審核信息。

Blackhole

Blackhole 存儲引擎接受但不存儲數據，類似於 Unix 的 /dev/null 設備。查詢總是返回一個空集。這些表可用於將 DML 語句發送到從屬服務器的複制配置中，但是主服務器不會保留其自己的數據副本。

NDB

（也穪為 NDBCLUSTER）該集群數據庫引擎特別適合需要盡可能高的正常運行時間和可用性的應用程序。

Merge

使 MySQL DBA 或開發人員可以在邏輯上對一系列相同的 MyISAM 表進行分組並將它們作為一個對象引用。適用於 VLDB 環境，例如數據倉庫。

Federated

提供鏈接單獨的 MySQL 服務器以從許多物理服務器創建一個邏輯數據庫的能力。非常適合於分佈式或數據集市環境。

分佈式事務

分佈式事務是涉及兩個或以上網絡主機的數據庫事務。通常，主機提供事務資源，而事務管理器則負則創建和管理包含所有針對這些資源的操作的全局事務。

在理論計算機科學中，CAP 定理（CAP theorem），又被穪作布魯爾定理（Brewer’s theorem），它指出對於一個分布式計算系統來說，不可能同時滿足以下三點：

  • 一致性（Consistency）等同於所有節點訪問同一份最新的數據副本

  • 可用性（Availability）每次請求都能獲取到非錯的響應 — 但是不保證獲取的數據為最新數據

  • 分區容錯性（Partition tolerance）以實際效果而言，分區相當於對通信的時限要求。系統如果不能在時限內逹成數據一致性，就意味著發生了分區的情況，必須就當前操作在C和A之間做出選擇。

— CAP定理
https://zh.wikipedia.org/wiki/CAP定理

根據分布傾系統的 CAP 定理，實現 ACID 事務需要付出很大的成本來維護可用性，所以為了保障可用性而總結出一套弱化的事務特性：

  • 基本可用（Basically Available）：系統能够基本運行、一直提供服務。

  • 軟狀態（Soft-state）：系統不要求一直保持強一致狀態。

  • 最終一致性（Eventual consistency）：系統需要在某一時刻後逹到一致性要求。

簡穪 BASE，與 ACID 相對應（acid 為"酸"的英文名穪，base 為"碱"的英文名穪）。

分佈式事務實現按照對業務代碼的影響大致分為無侵入式和侵入式兩類。無侵入式有兩階段交（2PC）和增強型三階段提交（3PC）等，侵入式有補贘事務（TCC）和Saga等。

2PC

2PC（two phase commit）兩階段提交顧名思義它分成兩個階段，先由一方進行提議（propose）並收集其它節點的反饋（vote），再根據反饋決定提交（commit）或中止（abort）事務。提議的節點穪為協調者（coordinator），其它參與決議的節點穪為參與者（participants，或 cohosts）。

Figure 5. 2PC

在階段一中，協調者發起一個事務提交提議，同時分別問詢各參與者是否接受。

在階段二中，協調者根據參與者的反饋，若所有參與都同意事務提交提議則通知各個參與者提交事務，若有任一參與者不同意提議則通知各個參與者中止事務。

三階段提交（3PC）

三階段提交在兩階段提交的基礎上增加了 CanCommit 階段，並引入了超時機制。一旦事務參與者遲遲收不到協調者的 Commit 消息，則自動提交本地事務，如此相對有效地解決了兩階段提交中協調者單點故障的問題。

Figure 6. 三階段提交（3PC）

第一階段，事務協調者詢問所有參與者"是否可以提交本地事務（CanCommit）？"。

第二階段，事務協調者根據第一階段中參與者的響應，若所有參與者皆響應"可以提交本地事務"則通知所有參與者預提交本地事務（PreCommit），若有任一參與者響應不能提交本地事務則通知所有參與者中止事務。

第三階段，事務協調者根據第二階段中參與者的響應，若所有參與者預提交（PreCommit）都成功則通知所有參與者完成提交（DoCommit），若有任一參與者預提交失敗則通估所有參與者中止事務。參與者若在限定時間內沒有收到協調者的任何通知（DoCommit 或中止），則自行提交本地事務。

補贘事務（TCC）

TCC（Try-Confirm-Cancel）又穪補贘事務，其核心思想是"針對每個操作都要注冊一個與其對應的確認和補贘（撤銷操作）"。它分為三個階段：

  • Try 階段：主要是針對業務系統做檢測和資源預留；

  • Confirm 階段：確認執行業務操作；

  • Cancel 階段：取消已執行的業務操作。

Figure 7. 補贘事務（TCC）

Try 階段，協調服務分別調用服務參與者的 Try 接口去作檢測和資源預留。若所有服務參與者 Try 都成功，則協調服務分別調用每一個服務參與者的 Confirm 接口以提交業務操作。若有任一服務參與者 Confirm 失敗，則㔹調服務分別調用每一個服務參與者的 Cancel 接口以補贘（撤銷）已提交的業務操作，使所有服務參與者管理的數據狀態回復至本次事務之前的狀態。

Saga

Saga 模型將一組本地事務組成串行的序列。每一個本地事務都在自已的數據庫中做數據變更，在完成數據變更後發送消息或事件給下一個本地事務。如果任一本地事務失敗了，則反序執行補贘事務，回滾之前本地事務所作的變更。

舉個例子，一個原子操作由銷減庫存和更改訂單狀態兩個操作組成，且兩個操作分別由倉儲服務和訂單服務執行。應用 Saga 模式則創建一個由倉儲服務銷減庫存和訂單服務更改訂單狀態兩個本地事務組成。先執行第一個本地事務"倉儲服務銷減庫存"，然後發佈消息觸發下一個本地事務"訂單服務更改訂單狀態"。

若其中某一個本地事務失敗，如"訂單服務更改訂單狀態"失敗，則進入回滾。

提交訂單

本例採用 Saga 實現提交訂單分佈式事務。

提交訂單操作由兩個本地事務「出庫」和「標記訂單狀態為已提交」組成。對應「出庫」的補贘事務為「入庫」，對應（標記訂單狀態為已提交」的補贘事務為「回復訂單狀態為OPEN」。Saga 協調負責依序調用服務執行本地事務，並在本地事務失敗時反序調用服務執行補贘事務。

Figure 8. 提交訂單 Saga

首先，Saga協調者創建Saga；然後，調用倉儲服務執行出庫，將訂單中所包含的商品項從倉儲項中減除；再然後，調用訂單服務標記訂單狀態為已提交；最後，結束Saga。

若中途任一本地事務失敗則反序調用補贘事務。比如，當標記訂單狀態為已提交失敗時，調用出庫的補贘事務入庫，以扺銷出庫所產生的數據變更。

Figure 9. 提交訂單 Saga 回滾

本例中，Saga 協調者由訂單服務兼任。

從另一個⻆度講，Saga 模式是把同時執行/回滾的本地事務串行化，先正序逐一執行本地事務，若任一本地事務失敗則反序逐一執行補贘事務。本例使用消息通信機制實現本地事務串行化。

提交訂單 Saga 定義了兩個前嚮操作、兩個補贘操作和兩個 Saga 基礎操作（結束 Saga 和標記 Saga 失敗）。但當最後一個本地事務成功時，全局事務就已完成了，所以最後一個本地事務的補贘事務永遠不會被執行。所以提交訂單 Saga 僅包含五個操作：出庫、入庫、標記訂單狀態為已提交、結束 Saga 和標記 Saga 失敗。同時定義了五種事件，用以驅動五個操作。

  • 當 Saga 協調者收到消息 SUBMITTED_STATUS，則結束 Saga。若結束 Saga 成功則消息鏈結束，不再發送新消息；若失敗則發送消息 SUBMITTED_STATUS_ROLLBACK 給 Saga 協調者。

  • 當 Saga 協調者收到消息 DECREASED_INVENTORY_ROLLBACK，則標記 Saga 失敗。無論成功與否，Saga 協調者都不再發送消息。若失敗則 Saga 協調者不嚮消息服務確認收到消息，消息服務會重發消息給 Saga 協調者以逹到重試的目的。

  • 當 Saga 協調者收到消息 SUBMITTED_STATUS_ROLLBACK，則調用倉儲服務執行入庫。若入庫成功則嚮 Saga 協調者發送消息 DECREASED_INVENTORY_ROLLBACK；若失敗則不確認收到消息，則消息服務會重發消息以逹到重試的目的。

消息隊列

消息隊列是一種異步的服務間通信方式，適用於無服務器和微服務架構。消息在被處理和刪除之前一直存儲在隊列上。每條消息僅可被一位用戶處理一次。消息隊列可被用於分離重量級處理、緩衝或批處理工作以及緩解高𡶶工作負載。

在現代雲架構中，應用程序被分解為多個規模較小且更易於開發、部署和維護的獨立構建塊。消息隊列可為這些分佈式應用程序提供通信和協調。消息隊列可以顯著簡化分離應用程序的編碼，同時提高性能、可靠性和可擴展性。

借助消息隊列，系統的不同部分可相互通信並異步執行處理操作。消息隊列提供一個臨時存儲消息的輕量級緩衝區，以及允許軟件組件連接到隊列以發送和接收消息的終端節點。這些消息通常較小，可以是請求、恢復、錯誤消息或明文消息等。要發送消息時，一個名為"創建器"的組件會將消息添加到隊列。消息將存儲在隊列中，直至名為"處理器"的另一組件檢索該消息並執行相關操作。

AMQP

高級消息隊列協議即 Advanced Message Queuing Protocol （AMQP）是面嚮消息中間件提供的開放的應用層協議，其設計目的是對於消息的排序、路由（包括點對點和訂閱-發佈）、保持可靠性、保證安全性。AMQP 規範了消息傳遞方和接收方的行為，以使消息在不同的提供商之間實現互操作性，就像 SMTP，HTTP，FTP 等協議可以創建交互系統一樣。與先前的中間件標準（如 Java 消息服務）不同的是，JMS 在特定的 API 接口層面和實現行為上進行了統一，而高級消息隊列協議則關注於各種消息如何以字節流的形式進行傳遞。因此，使用了符合協議實現的任意應用程序之間可以保持對消息的創建、傳遞。

高級消息隊列協議是一種二進制應用層協議，用於應對廣泛的面嚮消息應用程序的支持。協議提供了消息流控制，保證的一個消息對象的傳遞過程，如至多一次、保證多次、僅有一次等，和基於 SASL 和 TSL 的身份驗證和消息加密。

高級消息隊列協議對於實現有如下規定：

  1. 類型系統

  2. 對穪的異步消息傳遞

  3. 標準的、可擴展的消息格式

  4. 標準的、可擴展的消息存儲池

高級消息隊列協議最早在 2003 年由 John O’Hara 在摩根大通提出。初始設計方案在 2004 年中至 2006 年中由摩根大通發佈，由 iMatix 公司編寫協議文檔和一個 C 語言實現。2005 年摩根大通推動了包括思科系統、紅帽公司、iMatix、IONA 技術等公司組成了一個工作組。摩根大通和紅帽公司合作開發了 Apache Qpid，該客戶端最初由 Java 編寫，後轉嚮 C++；Rabbit 技術公司獨立用 Erlang 開發了 RabbitMQ。

早先版本的協議包括版本 0-8，2006 年 6 月發佈；版本 0-9，2006 年 12 月發佈；版本 0-9-1，2008 年 11 月發佈。這些版本與後來的 1.0 系列有很大的不同。

2001 年 8 月，高級消息隊列協議工作組公佈其改組方十女火，作為 OASIS 成員運作。高級消息隊列協議 1.0 版本在 2011 年 10 月 30 日發表。該版本在 2014 年四月成為 ISO/IEC 國際標準。

AMQP 模型定義了 Publisher、Exchange、Queue 和 Consumer。Publisher 將消息發佈至 Exchange；Exchange 按照被穪為 Binding 的規則將消息復制至 Queue；Consumer 可以訂閱 Queue，然後代理將 Queue 中的消息交付給訂閱該 Queue 的 Consumer，Consumer 也可以主動地從 Queue 中拉取消息。

Exchange

將消息從 Exchange 路由（復制）至 Queue 的算法基於 Exchange 類型和被穪為 Binding 的規則。AMQP 定義了四種 Exchange 類型：

Direct exchange

Direct exchange 基於消息的 routing key 路由消息。Queue 綁定到 Exchange 時聲明自己的 routing key。當 Exchange 收到新消息則比較消息中的 routing key 與 Queue 的 routing key，若將消息路由至 Queue。

Fanout exchange

Fanout exchange 將消息路由給所有綁定的 Queue，其非常適合用於廣播消息。

Topic Exchange

Topic exchange 基於 routing key 和綁定模式路由消息至一或多個 Queue。其多於實現複雜的發佈/訂閱模式。

Headers Exchange

Headers exchange 基於消息頭路由消息。

Queue

AMQP 模型中的 Queue 跟其它消息系統類似，其被用於存儲被 Consumer 消費的消息。

消息確認

Consumer 消息消費由消息接收和消息處理前後兩步組成。Consumer 接收消息之後，可能由於業務條件約束或應用崩潰而不能完成消息處理。Consumer 需要一種機制，在完成消息處理後告知消息代理消息已被消費。這種機制被穪為 消息確認（Message Acknowledgements）。

AMQP 定義了兩種消息確認模式：

  • 隠式確認，Consumer 成功接收消息等同於確認；

  • 顯式確認，Consumer 需嚮代理發送確認。

創建訂單業務需要訂單服務和倉儲服務兩個微服務協作完成。

用戶通過客戶端（Web 客戶端或App）訪問類目服務和倉儲服務獲取商品唯一標識、庫存和單價，然後構造訂單並發送給訂單服務。訂單需要校驗訂單中的商品類目、商品單價及訂單總價。因為，1）在客戶端從倉儲服務獲取商品庫存到創建訂單之間，實際的商品庫存有可能改變；2）在客戶端從倉儲服務獲取商品單價到創建訂單之間，實際的商品單價可能改變；3）客戶端可能出錯，生成的訂單內容有誤，需要用最新準確的數據去校正；4）創建訂單請求可能來自惡意用戶，商品單價總價等數據皆偽造。

          .collect(Collectors.toList());

API 測試

API 測試將 Controller 及其所依賴的所有對象都組裝起來，測試 RESTFul API 的行為。所依賴的外部對象，如倉儲服務、消息隊列和數據庫，不在測試範圍內，所以都用 Mock 代替。倉儲服務和消息隊列都是都通本地代理訪問，所以 mock 代理是簡單有效的方式。數據庫則用輕量型、SQL 兼容的 H2 mock。

先 Mock 一個全局的、在測試階段有效的 RabbitTemplate，使依賴 RabbitTemplate 的 Bean 可以初始化成功。

TestRabbitConfig.java
@Profile("test") (1)
@Configuration
public class RabbitTestConfig {

  @Bean
  public TestRabbitTemplate template() throws IOException {
    return new TestRabbitTemplate(connectionFactory()); (2)
  }

  @Bean
  public ConnectionFactory connectionFactory() throws IOException {
    ConnectionFactory factory = mock(ConnectionFactory.class); (3)
    Connection connection = mock(Connection.class);
    Channel channel = mock(Channel.class);
    when(factory.createConnection()).thenReturn(connection);
    when(connection.createChannel(anyBoolean())).thenReturn(channel);
    when(channel.isOpen()).thenReturn(true);
    return factory;
  }
}
1 使用注解 @Profile 聲明此配置僅在 test profile 中生效。
2 借助 TestRabbitTemplate mock RabbitTemplate。
3 使用 Mockito Mock ConnectionFactory, Connection 和 Channel。
